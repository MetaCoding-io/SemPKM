# Chapter 3: Installation and Setup

This chapter walks you through installing SemPKM, running the first-time setup wizard, configuring environment variables, and understanding the architecture of the three Docker services that make up the platform.

## Prerequisites

SemPKM runs as a set of Docker containers. Before you begin, make sure you have the following installed:

- **Docker** (version 20.10 or later)
- **Docker Compose** (version 2.0 or later, or the `docker compose` plugin)
- **Git** for cloning the repository

### System Requirements

| Resource | Minimum | Recommended |
|----------|---------|-------------|
| RAM | 2 GB | 4 GB |
| Disk | 1 GB | 5 GB |
| CPU | 1 core | 2 cores |

The triplestore (RDF4J) is configured with a 1 GB Java heap (`-Xmx1g`). If you plan to store large datasets, increase the `JAVA_OPTS` value in `docker-compose.yml`.

## Quick Start

### 1. Clone the Repository

```bash
git clone https://github.com/your-org/sempkm.git
cd sempkm
```

### 2. Start the Services

```bash
docker compose up
```

This single command builds the API container, pulls the RDF4J and nginx images, creates the Docker network and volumes, and starts all three services. The first run takes a few minutes while Docker builds the backend image and downloads dependencies.

You will see log output from all three services. Watch for the API startup messages -- they contain important information for the next step.

### 3. Access the Application

Once all services are healthy, open your browser and navigate to:

```
http://localhost:3000
```

The frontend is served by nginx on port 3000. If this is a fresh instance, you will be redirected to the **Setup** page to complete first-run configuration.

> **Tip:** If port 3000 is already in use on your machine, edit the `ports` mapping in `docker-compose.yml` under the `frontend` service. For example, change `"3000:80"` to `"8080:80"` and access the app at `http://localhost:8080`.

## First-Run Setup Wizard

On first startup, SemPKM detects that no owner account exists and enters **setup mode**. The API prints a one-time setup token to the console log.

### Step 1: Find the Setup Token

Look for this block in the API container's log output:

```
============================================================
  FIRST-RUN SETUP
  No owner found. Use this token to claim the instance:

  Setup token: <your-token-here>

  POST /api/auth/setup with {"token": "<token>"}
============================================================
```

The setup token is a random URL-safe string generated by the system and stored at `/app/data/.setup-token` inside the container. It is printed once at startup and remains valid until the instance is claimed.

> **Warning:** Anyone with access to this token can claim ownership of your SemPKM instance. Do not share it or commit it to version control.

### Step 2: Claim the Instance

Navigate to `http://localhost:3000/setup.html` in your browser. Enter the setup token from the logs and, optionally, your email address. Click **Claim Instance**.

<!-- Screenshot: Setup page with token input field and Claim Instance button -->

Behind the scenes, this sends a `POST /api/auth/setup` request with the token. The system:

1. Verifies the token matches the one generated at startup.
2. Creates the **owner** user account (using your email, or `owner@local` if you did not provide one).
3. Creates a session and sets an httpOnly cookie (`sempkm_session`) in your browser.
4. Deletes the setup token file from disk.
5. Disables setup mode -- the instance is now claimed.

After setup completes, you are automatically logged in as the owner and redirected to the main workspace.

### Step 3: Verify the Starter Model

On first startup, SemPKM automatically installs the **Basic PKM** Mental Model if no models are already present. You can verify this by checking the API logs for:

```
No models installed, auto-installing Basic PKM starter model
Basic PKM starter model installed successfully (model_id=basic-pkm)
```

The Basic PKM model provides the Note, Concept, Project, and Person types along with their shapes, views, and seed data. You should see seed objects (sample projects, people, notes, and concepts) already populated in the workspace.

## Authentication

SemPKM uses **passwordless authentication** via magic links. There are no passwords to manage.

### How Magic Links Work

1. On the login page (`/login.html`), enter your email address and click **Send Magic Link**.
2. The system generates a signed token valid for 10 minutes.
3. **If SMTP is configured:** The token is sent to your email as a login link. Click it to authenticate.
4. **If SMTP is not configured (default for local instances):** The token is returned directly in the browser and also printed to the API container logs. Copy and paste it into the verification field.

After verification, the system sets an httpOnly session cookie (`sempkm_session`) that persists for the duration configured by `SESSION_DURATION_DAYS` (default: 30 days).

### User Roles

SemPKM has two user roles:

- **Owner** -- Full access, including user management (inviting members) and admin operations. Created during the setup wizard.
- **Member** -- Standard access to create, edit, and browse objects. Created when invited by the owner or when a new email authenticates via magic link.

The owner can invite additional users from the workspace by sending invitations via `POST /api/auth/invite`. Invitation tokens are valid for 7 days.

## Configuration

SemPKM is configured through environment variables, which can be set in the `.env` file in the project root or directly in the `docker-compose.yml` `environment` section. The API reads these values at startup using Pydantic Settings.

### Environment Variables Reference

#### Triplestore Connection

| Variable | Default | Description |
|----------|---------|-------------|
| `TRIPLESTORE_URL` | `http://triplestore:8080/rdf4j-server` | URL of the RDF4J server API. Uses the Docker service name `triplestore` by default. |
| `REPOSITORY_ID` | `sempkm` | Name of the RDF4J repository to use. Created automatically on first startup if it does not exist. |
| `BASE_NAMESPACE` | `https://example.org/data/` | Base IRI namespace for user-created objects. Change this to your own domain in production (e.g., `https://yourcompany.com/kb/`). |

#### Database

| Variable | Default | Description |
|----------|---------|-------------|
| `DATABASE_URL` | `sqlite+aiosqlite:///./data/sempkm.db` | SQLAlchemy connection string for the SQL database. SQLite is used for local installations. PostgreSQL is supported for production deployments (e.g., `postgresql+asyncpg://user:pass@host/dbname`). |

The SQL database stores user accounts, sessions, and invitations. Object data (your actual knowledge) is stored in the RDF triplestore.

#### Security

| Variable | Default | Description |
|----------|---------|-------------|
| `SECRET_KEY` | (empty) | Secret key for signing tokens (magic links, invitations). If empty, SemPKM auto-generates a key and stores it at `./data/.secret-key` inside the container. Set this explicitly in production for predictable deployments. |
| `SESSION_DURATION_DAYS` | `30` | How long a session cookie remains valid, in days. |

> **Tip:** For local development, you can leave `SECRET_KEY` empty. SemPKM will generate one automatically and persist it in the data volume. For production deployments, set a strong explicit key to ensure sessions survive container restarts.

#### SMTP (Optional)

| Variable | Default | Description |
|----------|---------|-------------|
| `SMTP_HOST` | (empty) | SMTP server hostname. When empty, magic link tokens are returned directly in the browser. |
| `SMTP_PORT` | `587` | SMTP server port. |
| `SMTP_USER` | (empty) | SMTP authentication username. |
| `SMTP_PASSWORD` | (empty) | SMTP authentication password. |
| `SMTP_FROM_EMAIL` | (empty) | "From" address for outgoing emails. |

> **Note:** SMTP is optional. When not configured, magic link tokens are displayed directly in the browser and logged to the API terminal. This is convenient for local and single-user instances. Configure SMTP when you need email-based login for multi-user or remote deployments.

To enable SMTP, uncomment the SMTP variables in `docker-compose.yml` and provide your mail server credentials:

```yaml
environment:
  SMTP_HOST: smtp.example.com
  SMTP_PORT: 587
  SMTP_USER: your-username
  SMTP_PASSWORD: your-password
  SMTP_FROM_EMAIL: sempkm@example.com
```

#### Debugging

| Variable | Default | Description |
|----------|---------|-------------|
| `DEBUG` | `false` | Enables debug mode with additional logging. |

### The .env File

The project includes a `.env` file at the repository root with the three core settings:

```
TRIPLESTORE_URL=http://triplestore:8080/rdf4j-server
REPOSITORY_ID=sempkm
BASE_NAMESPACE=https://example.org/data/
```

Additional variables can be added to this file. Docker Compose automatically loads `.env` and interpolates its values into the service `environment` sections.

## Architecture Overview

SemPKM runs as three Docker services on a shared network called `sempkm`. Here is how they connect:

```
                         Port 3000
                            |
                     +------+------+
                     |   frontend  |
                     |   (nginx)   |
                     +------+------+
                            |
                       /api/* proxy
                            |
                     +------+------+
          Port 8001 -|     api     |
                     |  (FastAPI)  |
                     +------+------+
                            |
                    RDF4J HTTP API
                            |
                     +------+------+
                     | triplestore |
                     |  (RDF4J)    |
                     +-------------+
```

### Service: `triplestore` (Eclipse RDF4J)

The triplestore is the primary data store for all knowledge objects, relationships, events, model definitions, and validation reports. It runs Eclipse RDF4J 5.0.1, an open-source Java-based RDF database.

| Setting | Value |
|---------|-------|
| Image | `eclipse/rdf4j-workbench:5.0.1` |
| Internal port | 8080 |
| External port | None (not exposed to host) |
| Volume | `rdf4j_data` mounted at `/var/rdf4j` |
| Java heap | 1 GB (`-Xmx1g`) |
| Health check | `GET /rdf4j-server/protocol` every 10s |

On first startup, the API service automatically creates a repository named after the `REPOSITORY_ID` setting (default: `sempkm`) using a Turtle configuration file that specifies a native store with `cspo` indexes. A sentinel triple is inserted into the `urn:sempkm:current` named graph to prevent RDF4J from garbage-collecting the empty graph.

> **Note:** The triplestore is not exposed to the host by default. It is only accessible from within the Docker network. If you need direct access to the RDF4J Workbench for debugging, add a `ports` mapping like `"8080:8080"` to the `triplestore` service in `docker-compose.yml`.

### Service: `api` (FastAPI)

The API service is the application backend. It handles all business logic: authentication, object CRUD via the command system, SHACL validation, Mental Model management, view execution, and template rendering.

| Setting | Value |
|---------|-------|
| Built from | `./backend/Dockerfile` |
| Base image | `python:3.12-slim` |
| Internal port | 8000 |
| External port | 8001 (mapped from 8000) |
| Server | Uvicorn with auto-reload |
| Health check | `GET /api/health` every 10s |

The API container mounts four volumes:

| Mount | Container path | Purpose |
|-------|---------------|---------|
| `./backend/app` | `/app/app` | Application source code (live reload in development) |
| `./config` | `/app/config` (read-only) | RDF4J repository configuration files |
| `./models` | `/app/models` (read-only) | Mental Model packages available for installation |
| `sempkm_data` | `/app/data` | Persistent data: SQLite database, secret key, setup token |

The API depends on the triplestore being healthy before it starts. On startup, it:

1. Connects to the triplestore and ensures the RDF4J repository exists.
2. Creates the SQL database tables (users, sessions, invitations).
3. Auto-installs the Basic PKM starter model if no models are present.
4. Starts the background SHACL validation queue.
5. Detects whether setup has been completed and enters setup mode if needed.

> **Tip:** The API is also exposed directly on port 8001 for development and debugging. You can access the API docs at `http://localhost:8001/docs` (FastAPI's auto-generated OpenAPI documentation).

### Service: `frontend` (nginx)

The frontend service is a lightweight nginx reverse proxy that serves static assets and routes requests to the API.

| Setting | Value |
|---------|-------|
| Image | `nginx:stable-alpine` |
| Internal port | 80 |
| External port | 3000 |
| Depends on | `api` (healthy) |

Nginx handles the following routing:

| Path pattern | Destination |
|-------------|-------------|
| `/css/*`, `/js/*` | Served directly from static files (no cache headers for development) |
| `/setup.html`, `/login.html`, `/invite.html` | Served as static HTML files |
| `/api/*` | Proxied to `http://api:8000/api/` with cookie forwarding |
| Everything else (`/`) | Proxied to `http://api:8000/` for server-side template rendering |

The proxy configuration forwards cookies (`Cookie` and `Set-Cookie` headers) to support the httpOnly session cookie authentication. It also passes standard proxy headers (`X-Real-IP`, `X-Forwarded-For`, `X-Forwarded-Proto`) for accurate logging and security.

### Docker Volumes

SemPKM uses two named Docker volumes for persistent storage:

| Volume | Mounted at | Contents |
|--------|-----------|----------|
| `rdf4j_data` | `/var/rdf4j` (triplestore) | All RDF data: objects, events, model graphs, validation reports |
| `sempkm_data` | `/app/data` (api) | SQLite database (users, sessions), auto-generated secret key, setup token |

> **Warning:** Deleting these volumes will permanently erase all your data. Use `docker compose down` (without the `-v` flag) to stop services while preserving data. Only use `docker compose down -v` when you intentionally want to reset to a clean state.

### Docker Network

All three services communicate over a bridge network named `sempkm`. The service names (`triplestore`, `api`, `frontend`) serve as DNS hostnames within this network. This is why the default `TRIPLESTORE_URL` uses `http://triplestore:8080/rdf4j-server` -- the `triplestore` hostname resolves to the triplestore container's IP address within the Docker network.

## Stopping and Restarting

To stop all services while preserving your data:

```bash
docker compose down
```

To restart:

```bash
docker compose up
```

The API will detect that setup has already been completed (the owner account exists in the SQL database) and skip setup mode. Your data in both the triplestore and SQL database persists across restarts.

To completely reset the instance (delete all data and start fresh):

```bash
docker compose down -v
docker compose up
```

> **Warning:** The `-v` flag removes all named volumes, permanently deleting your RDF data and user accounts. The setup wizard will appear again on next startup.

## Verifying Your Installation

After completing the setup wizard, verify that everything is working:

1. **Check the workspace** -- Navigate to `http://localhost:3000`. You should see the IDE-style workspace with a navigation tree on the left showing the Basic PKM types (Notes, Concepts, Projects, People).

2. **Browse seed data** -- Click on **Projects** in the navigation tree. You should see the seed projects ("SemPKM Development" and "Knowledge Garden") in the default table view.

3. **Open an object** -- Click on "SemPKM Development" to open it in the editor. You should see the auto-generated form with fields for Title, Description, Status, Priority, Start Date, Tags, and relationships to participants and notes.

4. **Check the health endpoint** -- Visit `http://localhost:8001/api/health` in your browser or run:

   ```bash
   curl http://localhost:8001/api/health
   ```

   A healthy response confirms the API is running and connected to the triplestore.

5. **Check the API docs** -- Visit `http://localhost:8001/docs` for the interactive OpenAPI documentation.
