# T02: Create dual-instance docker-compose and lifecycle scripts

**Created a 6-service docker-compose and 3 lifecycle scripts that stand up two complete SemPKM stacks for federation testing, verified with a full start → health check → stop cycle.**

## What Happened

Created `docker-compose.federation-test.yml` with 6 services: two complete SemPKM stacks (triplestore, API, nginx frontend) on a shared `sempkm-federation` Docker network. Each instance has its own `BASE_NAMESPACE`, `APP_BASE_URL` (set to Docker service name for cross-instance resolution), `SECRET_KEY`, RDF4J triplestore, and SQLite database volume. Port mappings are 3911/8911 for A and 3912/8912 for B, avoiding conflicts with dev and test stacks.

The nginx frontend containers required per-instance configs because the shared `nginx.conf` references `proxy_pass http://api:8000` but the federation compose has `api-a` and `api-b` as service names. Solved by generating `nginx-a.conf` and `nginx-b.conf` via `sed` substitution from the main config. The start script regenerates these on each run so they stay in sync.

Created three lifecycle scripts matching the existing `e2e/scripts/` pattern:
- `start-federation-env.sh` — generates nginx configs, tears down, builds, starts, waits
- `wait-for-federation.sh` — polls both health endpoints with linear backoff (cap 5s, max 20 attempts)
- `stop-federation-env.sh` — tears down containers, volumes, and network

## Deviations

- **Added per-instance nginx configs** — The task plan didn't anticipate that `frontend/nginx.conf` hardcodes `proxy_pass http://api:8000` which fails when the API service is named `api-a`/`api-b`. Created `e2e/federation/nginx-a.conf` and `nginx-b.conf` via sed substitution, regenerated by `start-federation-env.sh` on each run.
- **Added `e2e/federation/` directory** — New directory to hold generated federation test configs (nginx configs now, potentially more later).
