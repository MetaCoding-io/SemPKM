  <action>
    1. Create `backend/app/templates/forms/_field.html` with a Jinja2 macro that dispatches form fields by property shape attributes (per Research Jinja2 form field macro pattern):

       `{% macro render_field(prop, value=None, values=None) %}`

       Dispatch logic:
       - **sh:in constraints** (prop.in_values is not empty):
         * If max_count == 1 or None: standard `<select>` dropdown with options from in_values
         * If max_count > 1: multi-select or checkboxes
         * Pre-select current value or defaultValue
       - **sh:class references** (prop.target_class is set):
         * Search-as-you-type text input with `hx-get="/browser/search?type={{ prop.target_class | urlencode }}&q="` `hx-trigger="input changed delay:300ms"` `hx-target` pointing to a suggestions dropdown div
         * Include "Create new..." option at bottom of suggestions (per CONTEXT.md decision)
         * Display selected reference as label + hidden input with IRI value
       - **xsd:date**: `<input type="date">`
       - **xsd:dateTime**: `<input type="datetime-local">`
       - **xsd:boolean**: `<select>` with Yes/No options
       - **xsd:integer**: `<input type="number" step="1">`
       - **xsd:decimal / xsd:float / xsd:double**: `<input type="number" step="0.01">`
       - **xsd:anyURI**: `<input type="url">`
       - **Default (xsd:string or no datatype)**: `<input type="text">`

       Each field wrapper:
       - `<div class="form-field {required if minCount > 0}" id="field-{prop.path encoded}">`
       - Label: `<label title="{{ prop.path }}">{{ prop.name }}{% if prop.min_count > 0 %}<span class="required-marker">*</span>{% endif %}</label>`
       - The `title` attribute shows full IRI on hover (per CONTEXT.md: "hovering reveals the full URI/prefixed name as a tooltip")
       - Description: `{% if prop.description %}<small class="field-help">{{ prop.description }}</small>{% endif %}`
       - Validation message area: `<div class="field-validation" id="validation-{{ prop.path encoded }}"></div>`

       Multi-valued properties (prop.max_count is None or > 1):
       - Render existing values as a list, each with a remove button
       - "+" add button below to add another value
       - Use htmx or vanilla JS for dynamic add/remove (no page reload)
       - Name attribute uses `prop.path` with array suffix for multi-values

       Default value styling: if value is None and prop.default_value exists, pre-fill and add class `default-value` (lighter text per CONTEXT.md decision)

    2. Create `backend/app/templates/forms/_group.html`:
       - `{% macro render_group(group, properties, values) %}`
       - Renders as a `<details open>` element (collapsible, expanded by default per CONTEXT.md)
       - Summary: group label
       - Content: iterate over properties in this group, call `render_field` for each

    3. Create `backend/app/templates/forms/object_form.html`:
       - Import macros from `_field.html` and `_group.html`
       - Accepts: `form` (NodeShapeForm), `values` (dict of property path -> current value(s)), `mode` ("create" or "edit"), `object_iri` (None for create)
       - Form structure:
         * `<form hx-post="/browser/objects" hx-target="#editor-area" hx-swap="innerHTML">`
           (for create mode; edit mode uses different action URL)
         * Hidden input: `type_iri` (the target class)
         * Hidden input: `object_iri` (for edit mode)
         * Required fields first (minCount > 0), sorted by sh:order
         * Then grouped properties rendered via _group.html macro
         * Optional fields in a collapsible "Advanced" section (per CONTEXT.md: "optional fields collapsed in an Advanced section")
         * Submit button: "Create {type_label}" or "Save Changes"

    4. Create `frontend/static/css/forms.css`:
       - Form field layout: label above input, full width, consistent spacing
       - Required marker: red asterisk
       - Default value text: lighter color (e.g., #999)
       - Collapsible group: details/summary styling with arrow indicator
       - Advanced section: collapsed by default, lighter border
       - Dropdown suggestions: absolute positioned below input, z-index above form
       - Multi-value list: remove buttons inline, add button below
       - Field validation area: red text for errors, yellow for warnings
       - Overall clean, professional form styling matching IDE aesthetic
  </action>

<task type="auto">
  <name>Task 2: Type picker, create object flow, and edit object flow</name>
  <files>

  <action>
    1. Create `backend/app/templates/browser/type_picker.html`:
       - Modal/dialog overlay listing all available object types from installed models
       - Each type shown as a card or list item with: type label, description (from shape or ontology)
       - Clicking a type: `hx-get="/browser/objects/new?type={{ type_iri | urlencode }}"` `hx-target="#editor-area"` `hx-swap="innerHTML"`
       - This opens the create form in the editor area and adds a "New {Type}" tab
       - Also accessible from command palette "New Object" action (wire the showTypePicker() placeholder from Plan 04)

    2. Add form and object management routes to `backend/app/browser/router.py`:

       **Type picker:**
       - `GET /browser/types` -> render `browser/type_picker.html` with types from ShapesService.get_types()

       **Create flow:**
       - `GET /browser/objects/new?type={type_iri}` -> render create form:
         * Get form metadata from `shapes_service.get_form_for_type(type_iri)`
         * Render `forms/object_form.html` with mode="create", empty values, type_iri
         * Return as htmx partial (content for editor area)
       - `POST /browser/objects` -> process create form submission:
         * Parse form data: extract type_iri and property values from form fields
         * Build an `object.create` command payload with properties dict
         * Dispatch command internally (import and call the command handler directly, or POST to `/api/commands` internally using httpx, or use the command dispatcher from the commands module)
         * On success: redirect/load the created object in edit mode (render its object tab)
         * On error: re-render form with error messages

       **Edit flow:**
       - `GET /browser/objects/{object_iri}` -> render object in editor:
         * Query current property values from triplestore (SPARQL SELECT on urn:sempkm:current for the object)
         * Get the object's type(s) from rdf:type triple
         * Get form metadata from ShapesService for that type
         * Render `forms/object_form.html` with mode="edit", current values, object_iri
         * This is what opens when clicking an object in the nav tree
       - `PUT /browser/objects/{object_iri}` or `POST /browser/objects/{object_iri}/save` -> process edit form submission:
         * Parse form data and compare with current values to detect changes
         * Build `object.patch` command(s) for changed properties
         * Dispatch commands
         * Re-render the form with updated values and success message
         * Mark tab as clean (trigger JS event for tab management)

       **Reference search (for sh:class fields):**
       - `GET /browser/search?type={type_iri}&q={query}` -> search instances of a type:
         * SPARQL query on urn:sempkm:current for objects matching rdf:type and FILTER regex on labels
         * Use LabelService.resolve_labels() for display names
         * Return HTML list items for the suggestions dropdown
         * Include "Create new {type}..." option at the bottom

    3. Wire the type picker into the workspace:
       - Update workspace.js `showTypePicker()` to use htmx: `htmx.ajax('GET', '/browser/types', {target: '#editor-area', swap: 'innerHTML'})`
       - When a type is selected and form loads, create a "New {Type}" tab in the tab bar
       - After successful creation, update the tab with the real object IRI and label
  </action>
