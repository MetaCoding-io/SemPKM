2. **backend/app/commands/schemas.py**: Pydantic v2 models using discriminated unions per user decision and research pattern:

   **Parameter models:**
   - `ObjectCreateParams(BaseModel)`: `type: str` (required, the RDF type local name), `slug: str | None = None`, `properties: dict[str, Any] = {}` (predicate -> value pairs in JSON-LD compact form)
   - `ObjectPatchParams(BaseModel)`: `iri: str` (required, the object IRI), `properties: dict[str, Any]` (predicate -> new value pairs; only specified predicates are updated)
   - `BodySetParams(BaseModel)`: `iri: str` (required), `body: str` (Markdown content)
   - `EdgeCreateParams(BaseModel)`: `source: str` (source object IRI), `target: str` (target object IRI), `predicate: str` (relationship type IRI), `properties: dict[str, Any] = {}` (optional edge annotations)
   - `EdgePatchParams(BaseModel)`: `iri: str` (edge resource IRI), `properties: dict[str, Any]` (updated edge annotations)

   **Command models (discriminated union):**
   - `ObjectCreateCommand(BaseModel)`: `command: Literal["object.create"]`, `params: ObjectCreateParams`
   - `ObjectPatchCommand(BaseModel)`: `command: Literal["object.patch"]`, `params: ObjectPatchParams`
   - `BodySetCommand(BaseModel)`: `command: Literal["body.set"]`, `params: BodySetParams`
   - `EdgeCreateCommand(BaseModel)`: `command: Literal["edge.create"]`, `params: EdgeCreateParams`
   - `EdgePatchCommand(BaseModel)`: `command: Literal["edge.patch"]`, `params: EdgePatchParams`

   **Union type:**
   ```python
   Command = Annotated[
       Union[ObjectCreateCommand, ObjectPatchCommand, BodySetCommand, EdgeCreateCommand, EdgePatchCommand],
       Field(discriminator="command"),
   ]
   ```

   **Request model:**
   - `CommandRequest(BaseModel)`: Accept both single and batch: use a field validator or custom type so `{"command": "object.create", "params": {...}}` (single) and `[{"command": "object.create", ...}, {"command": "edge.create", ...}]` (batch) both work. Normalize to always return a list internally.

   **Response models:**
   - `CommandResult(BaseModel)`: `iri: str`, `event_iri: str`, `command: str`
   - `CommandResponse(BaseModel)`: `results: list[CommandResult]`, `event_iri: str`, `timestamp: str`

1. **backend/app/commands/handlers/__init__.py**: Empty init.

2. **Each handler** follows the same pattern: takes params + base_namespace, returns an `Operation` dataclass (from events/store.py). Handlers do NOT call the event store directly -- they produce Operations that the router collects and commits as a batch.

3. **object_create.py** - `async def handle_object_create(params: ObjectCreateParams, base_namespace: str) -> Operation`:
   - Mint object IRI via `mint_object_iri(base_namespace, params.type, params.slug)`
   - Build data triples: `(subject, RDF.type, URIRef(type_iri))` where type_iri is constructed from base_namespace + params.type
   - For each property in params.properties: resolve predicate IRI (if compact, expand using known prefixes; if already a full IRI, use as-is), create triple `(subject, predicate, value_literal_or_uri)`
   - Return `Operation(operation_type="object.create", affected_iris=[object_iri], description=f"Created {params.type}: {params.slug or 'new'}", data_triples=triples, materialize_inserts=triples, materialize_deletes=[])`

4. **object_patch.py** - `async def handle_object_patch(params: ObjectPatchParams, base_namespace: str) -> Operation`:
   - Use `params.iri` as the subject
   - For each property: build delete pattern `(subject, predicate, Variable("old"))` and insert triple `(subject, predicate, new_value)`
   - Return Operation with both `materialize_inserts` and `materialize_deletes`. The data_triples in the event graph should record the new values (what was set).
   - Note: materialize_deletes are triple patterns with Variables that will be used in DELETE WHERE clauses

5. **body_set.py** - `async def handle_body_set(params: BodySetParams, base_namespace: str) -> Operation`:
   - Use `params.iri` as subject
   - Use `SEMPKM.body` as the predicate for Markdown content
   - Delete old body (pattern), insert new body (literal with xsd:string datatype)
   - Return Operation

6. **edge_create.py** - `async def handle_edge_create(params: EdgeCreateParams, base_namespace: str) -> Operation`:
   - Mint edge IRI via `mint_edge_iri(base_namespace)` (edges are first-class resources with own IRIs per user decision)
   - Build triples: `(edge, RDF.type, SEMPKM.Edge)`, `(edge, SEMPKM.source, source_iri)`, `(edge, SEMPKM.target, target_iri)`, `(edge, SEMPKM.predicate, predicate_iri)`
   - For each property in params.properties: add annotation triples on the edge resource
   - Return Operation with affected_iris including both the edge IRI and the source/target IRIs

7. **edge_patch.py** - `async def handle_edge_patch(params: EdgePatchParams, base_namespace: str) -> Operation`:
   - Similar to object_patch but operates on edge resource IRI
   - Only updates annotation properties, NOT source/target/predicate (those are immutable)
   - Return Operation with deletes and inserts

8. **backend/app/commands/router.py** - `POST /api/commands` endpoint:
   - Accept the request body. Handle both a single command object and an array (batch). Per user decision: RPC-style single endpoint, JSON-LD payloads, batch support, all-or-nothing transactions.
   - Parse into list of Command objects via CommandRequest
   - For each command: call `dispatch(command, settings.base_namespace)` to get an Operation
   - Collect all Operations into a list
   - Call `event_store.commit(operations)` which handles the single transaction
   - Return `CommandResponse` with results
   - On any `CommandError`: return appropriate HTTP status code with error message
   - On transaction failure: the event store handles rollback; return 500
   - Register router on the FastAPI app in main.py (add `app.include_router(commands_router)`)
