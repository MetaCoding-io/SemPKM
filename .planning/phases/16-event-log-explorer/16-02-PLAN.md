**Filter chips row**: Each active filter renders as a button that, when clicked, reloads the event log without that filter param. Construct the "remove" URL by taking `current_params` dict, removing the key for that chip, and encoding with `urlencode()`:

```html
<div class="filter-chips" id="event-filter-chips">
  {% for chip in active_filters %}
  <button class="filter-chip"
          hx-get="/browser/events?{{ current_params | dict_without(chip.param) | urlencode }}"
          hx-target="#panel-event-log"
          hx-swap="innerHTML">
    {{ chip.label }} <span aria-hidden="true">&times;</span>
  </button>
  {% endfor %}
</div>
```

**`dict_without` Jinja2 filter**: Register a custom Jinja2 filter in the project's template environment that removes a key from a dict and returns the modified dict. Find where Jinja2 filters are registered in the project (look in `main.py` or `app/templates.py` or wherever `Jinja2Templates` is instantiated). Add:
```python
def dict_without(d: dict, key: str) -> dict:
    return {k: v for k, v in d.items() if k != key}

templates.env.filters["dict_without"] = dict_without
```

Also register a `urlencode` filter if not already present (Jinja2's built-in `urlencode` works for dicts in some versions; `urllib.parse.urlencode` is safer):
```python
from urllib.parse import urlencode as _urlencode
templates.env.filters["urlencode"] = lambda d: _urlencode(d) if isinstance(d, dict) else str(d)
```

Check if `urlencode` is already available in the project's Jinja2 env (search templates for existing `| urlencode` usage). If the existing codebase already registers it, skip.

Find the templates instantiation location and add these filters there.

**Click-to-filter on object and user links**: In the event row template (created in Plan 16-01), add htmx attributes to `.event-obj-link` and `.event-user` elements so clicking them filters the timeline by that object IRI or user respectively:

**Filter controls dropdown**: Below the filter chips, add operation type filter (and secondary filters for date range):

```html
<div class="event-filter-controls">
  <select class="event-filter-select" id="event-op-filter"
          hx-get="/browser/events"
          hx-target="#panel-event-log"
          hx-swap="innerHTML"
          hx-include="#event-date-from,#event-date-to"
          name="op"
          hx-trigger="change">
    <option value="">All operations</option>
    <option value="object.create" {% if current_params.get('op') == 'object.create' %}selected{% endif %}>object.create</option>
    <option value="object.patch" {% if current_params.get('op') == 'object.patch' %}selected{% endif %}>object.patch</option>
    <option value="body.set" {% if current_params.get('op') == 'body.set' %}selected{% endif %}>body.set</option>
    <option value="edge.create" {% if current_params.get('op') == 'edge.create' %}selected{% endif %}>edge.create</option>
    <option value="edge.patch" {% if current_params.get('op') == 'edge.patch' %}selected{% endif %}>edge.patch</option>
  </select>
  <input type="date" class="event-filter-date" id="event-date-from" name="date_from"
         value="{{ current_params.get('date_from', '') }}"
         hx-get="/browser/events" hx-target="#panel-event-log" hx-swap="innerHTML"
         hx-include="#event-op-filter,#event-date-to" hx-trigger="change"
         placeholder="From">
  <input type="date" class="event-filter-date" id="event-date-to" name="date_to"
         value="{{ current_params.get('date_to', '') }}"
         hx-get="/browser/events" hx-target="#panel-event-log" hx-swap="innerHTML"
         hx-include="#event-op-filter,#event-date-from" hx-trigger="change"
         placeholder="To">
</div>
```

NOTE: The `hx-include` selector approach targets elements by ID to combine filter values. Since the panel re-renders `#panel-event-log` (which contains all the filter controls), the IDs remain stable across reloads. This is the correct htmx pattern for combining multiple filter inputs without a form wrapper.

NOTE: The `op` dropdown with value `""` means "no filter" — when `op=` is submitted with empty string, the Python route receives `op=None` (FastAPI strips empty string query params as None for `str | None` typed params). Verify this behavior; if FastAPI does not strip empty strings, add logic in the route: `if op == "": op = None`.

**Load More button fix**: The Load More button in Plan 16-01 used `hx-select=".event-row"` to only append event rows. Replace this with a cleaner approach: wrap the paginated content in a sentinel div. When Load More is clicked, it should append new `.event-row` elements to `#event-timeline`. The correct htmx pattern:

```html
{% if next_cursor %}
<div class="event-load-more">
  <button class="btn-load-more"
          hx-get="/browser/events?cursor={{ next_cursor | urlencode }}{% for f in active_filters %}&{{ f.param }}={{ f.value | urlencode }}{% endfor %}"
          hx-target="#event-timeline"
          hx-swap="beforeend"
          hx-select=".event-row">
    Load more
  </button>
</div>
{% endif %}
```

Note: `hx-select=".event-row"` combined with `hx-swap="beforeend"` on `#event-timeline` is the correct htmx pattern to append only the event rows from the response into the existing timeline. The response will be a full `event-log-container` but only `.event-row` elements will be inserted.
