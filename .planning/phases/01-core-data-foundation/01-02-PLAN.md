   - `EVENT_TYPE = SEMPKM.Event` (rdf:type for events)
   - `EVENT_TIMESTAMP = SEMPKM.timestamp`
   - `EVENT_OPERATION = SEMPKM.operationType`

   - `EVENT_DESCRIPTION = SEMPKM.description`
   - `STATE_GRAPH_TYPE = SEMPKM.StateGraph`

3. **backend/app/events/store.py**: `EventStore` class:
   - Constructor takes a `TriplestoreClient` instance
   - **Core method: `async def commit(operations: list[Operation]) -> EventResult`** where `Operation` is a dataclass:
     ```python
     @dataclass
     class Operation:
         operation_type: str          # e.g. "object.create"
         affected_iris: list[str]     # IRIs changed
         description: str             # Human-readable
         data_triples: list[tuple]    # (s, p, o) triples to add to event graph
         materialize_inserts: list[tuple]  # Triples to INSERT into current state
         materialize_deletes: list[tuple]  # Triple patterns to DELETE from current state (optional)
     ```
   - `commit()` implementation:
     a. Mint a new event IRI via `mint_event_iri()`
     b. Begin an RDF4J transaction via `client.begin_transaction()`
     c. Build event metadata triples: type, timestamp (UTC ISO), combined operation types, all affected IRIs, description
     d. Build SPARQL INSERT DATA for the event named graph: metadata triples + all data_triples from all operations
     e. Execute via `client.transaction_update(txn_url, sparql)`
     f. For each operation's `materialize_inserts`: build SPARQL INSERT DATA into `<urn:sempkm:current>` graph, execute via transaction_update
     g. For each operation's `materialize_deletes`: build SPARQL DELETE ... WHERE against `<urn:sempkm:current>` graph, execute via transaction_update
     h. Commit the transaction via `client.commit_transaction(txn_url)`
     i. On any error: rollback via `client.rollback_transaction(txn_url)` and raise
     j. Return `EventResult(event_iri=..., timestamp=..., affected_iris=[...])`
   - **CRITICAL**: Event graph creation AND current state materialization MUST be in the SAME RDF4J transaction (per research Pitfall 2). Do NOT commit between event write and materialization.
   - **CRITICAL**: Use `GRAPH <urn:sempkm:current>` clause in all materialization SPARQL (per research Pitfall 3 -- scope to current state graph).
   - Helper `_serialize_rdf_term(term) -> str` for building SPARQL: handle URIRef (wrap in <>), Literal (with datatype/language), BNode.
   - Helper `_build_insert_data_sparql(graph_iri: URIRef, triples: list[tuple]) -> str` for constructing INSERT DATA with GRAPH clause.
   - Helper `_build_delete_insert_sparql(graph_iri: URIRef, deletes: list[tuple], inserts: list[tuple]) -> str` for constructing DELETE/INSERT/WHERE patterns for patches.
  </action>
