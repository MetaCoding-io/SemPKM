    Create `e2e/tests/16-indieauth/indieauth-flow.spec.ts` covering the full IndieAuth flow.

    **Test structure** (use existing e2e patterns: import from `@fixtures/auth`, `@helpers/selectors`, `@helpers/wait-for`):

    1. **Test: metadata endpoint returns valid IndieAuth server metadata**
       - GET `/api/indieauth/metadata`
       - Assert response is JSON with: issuer, authorization_endpoint, token_endpoint, introspection_endpoint, code_challenge_methods_supported includes "S256", scopes_supported includes "profile".

    2. **Test: well-known redirect points to metadata**
       - GET `/.well-known/oauth-authorization-server` (with redirect: "manual")
       - Assert 307 redirect with Location header pointing to `/api/indieauth/metadata`.

    3. **Test: profile page includes indieauth-metadata link**
       - Requires a user with published WebID profile.
       - First, set up a user with username and published profile (use API calls from test).
       - GET `/users/{username}` with Accept: text/html
       - Assert response contains `<link rel="indieauth-metadata"`.
       - Also check Link header contains `rel="indieauth-metadata"`.

    4. **Test: full authorization code flow with PKCE**
       - This is the core integration test. Simulate what an IndieAuth client does:
       - a. Generate PKCE code_verifier (random 43-char string) and code_challenge (SHA-256 + base64url no padding).
       - b. Log in as a user (use `loginViaApi()` fixture pattern).
       - c. GET `/api/indieauth/authorize?response_type=code&client_id=https://example.com/&redirect_uri=https://example.com/callback&scope=profile&code_challenge={challenge}&code_challenge_method=S256&state=teststate`.
       - d. Assert consent page is shown with app name and scope info.
       - e. Submit the consent form with action=approve (POST to `/api/indieauth/authorize` with form data).
       - f. Assert redirect to `https://example.com/callback` with `code`, `state=teststate`, and `iss` parameters in the URL.
       - g. Extract the `code` from the redirect URL.
       - h. POST `/api/indieauth/token` with form-encoded body: grant_type=authorization_code, code={code}, client_id=https://example.com/, redirect_uri=https://example.com/callback, code_verifier={verifier}.
       - i. Assert response JSON has access_token, token_type=Bearer, scope=profile, me (profile URL).
       - j. Extract access_token and refresh_token.

    5. **Test: token introspection**
       - Use the access_token from test 4.
       - POST `/api/indieauth/introspect` with form-encoded: token={access_token}.
       - Assert response has active=true, me, client_id, scope.
       - POST with invalid token: assert active=false.

    6. **Test: consent screen shows when user is authenticated**
       - Verify the consent screen renders correctly with scope descriptions.

    7. **Test: deny returns error to client**
       - Submit consent form with action=deny.
       - Assert redirect to redirect_uri with error=access_denied.

    **PKCE helper in test file:**
    ```typescript
    import { createHash, randomBytes } from 'crypto';

    function generatePKCE() {
      const verifier = randomBytes(32).toString('base64url');
      const challenge = createHash('sha256').update(verifier).digest('base64url');
      return { verifier, challenge };
    }
    ```

    **Important test notes:**
    - For redirect assertion: the authorize POST will redirect to an external URL (example.com). Use `request.newContext({ baseURL: ... })` or intercept the redirect response. Since Playwright follows redirects by default, use `page.route()` to intercept the redirect to example.com and capture the URL, OR use `request.fetch()` with redirect: "manual" to get the 302 response.
    - Use the API-level `request` context for token/introspection tests (no browser needed).
    - For consent page tests, use Playwright page navigation since it renders HTML.
