- Add a new private method `_load_rules_graphs(self) -> Graph` following the same pattern as `_load_ontology_graphs()`:
  - SPARQL query to list installed models from `urn:sempkm:models`
  - For each model, CONSTRUCT all triples from `urn:sempkm:model:{modelId}:rules`
  - Merge into a single rdflib Graph
  - Return the merged graph
- Modify `run_inference()` to add SHACL-AF rules step between step 4 (OWL 2 RL closure) and step 5 (diff). Insert after line `owlrl.DeductiveClosure(owlrl.OWLRL_Semantics).expand(working)`:
  ```python
  # 4b. Run SHACL-AF rules (if enabled)
  if entailment_config.get("sh:rule", False) or entailment_config.get("shacl_rules", False):
      rules_graph = await self._load_rules_graphs()
      if len(rules_graph) > 0:
          pre_rules = set(working)
          try:
              expanded = await asyncio.to_thread(
                  pyshacl.shacl_rules,
                  working,
                  shacl_graph=rules_graph,
                  advanced=True,
                  iterate_rules=True,
              )
              # shacl_rules returns the expanded graph
              rule_new = set(expanded) - pre_rules
              for t in rule_new:
                  working.add(t)
              logger.info("SHACL rules produced %d new triples", len(rule_new))
          except TypeError:
              # iterate_rules may not be supported -- fall back to single pass
              expanded = await asyncio.to_thread(
                  pyshacl.shacl_rules,
                  working,
                  shacl_graph=rules_graph,
                  advanced=True,
              )
              rule_new = set(expanded) - pre_rules
              for t in rule_new:
                  working.add(t)
              logger.info("SHACL rules (single pass) produced %d new triples", len(rule_new))
  ```
- **Critical per Pitfall 2**: After the existing classify step (step 8), rule-derived triples that return `None` from `classify_entailment()` will be dropped. To fix this, track which triples came from SHACL rules. Before calling `classify_entailment`, check if the triple is in the `rule_new` set; if so, tag it directly as `"sh:rule"` instead of calling `classify_entailment()`. Implementation approach:

  - In the classification loop (step 8), check `if (s, p, o) in rule_new_triples` first; if yes, append with `"sh:rule"` type; otherwise, call `classify_entailment()` as before.
