<!-- Ontology SPARQL to extract concrete examples -->
For owl:inverseOf:
SELECT ?p ?q WHERE { ?p owl:inverseOf ?q } (from model ontology graph)
→ shows "hasParticipant ↔ participatesIn"

For rdfs:subClassOf:
SELECT ?sub ?super WHERE { ?sub rdfs:subClassOf ?super . FILTER(!isBlank(?sub)) FILTER(!isBlank(?super)) }

For owl:TransitiveProperty:
SELECT ?p WHERE { ?p a owl:TransitiveProperty }

For rdfs:domain/rdfs:range:
SELECT ?p ?domain ?range WHERE { OPTIONAL { ?p rdfs:domain ?domain } OPTIONAL { ?p rdfs:range ?range } FILTER(BOUND(?domain) || BOUND(?range)) }

     - Save to SettingsService as `inference.{model_id}.{type}` = "true"/"false"
     - Return success response (htmx swap or redirect)

1. **Modify model uninstall in `backend/app/models/registry.py`** (or wherever model uninstall/removal logic lives):
   - After the existing model uninstall cleanup (removing model graphs, shapes, data), add a step to clear inferred triples:
     - Execute `DROP GRAPH <urn:sempkm:inferred>` (since inferred triples may reference the uninstalled model's ontology axioms, the safest approach is to drop all inferred data and let the user re-run inference after uninstall)
     - Clean up `inference_triple_state` SQLite records (delete all rows, since the inferred graph was dropped)
     - Clean up entailment settings for this model from SettingsService (delete keys like `inference.{model_id}.*`)

2. **Add cleanup to admin uninstall route in `backend/app/admin/router.py`**: If the uninstall is triggered from the admin UI, ensure the cleanup above is called. Follow the existing uninstall flow — the cleanup should be part of the same transaction/operation.

   Note: Dropping the entire inferred graph on model uninstall is the correct approach because:
   - Inferred triples may cross-reference multiple models' ontology axioms
   - Selective removal (only triples from model X) would require provenance tracking we don't have
   - Full recompute after uninstall (user clicks Refresh) rebuilds the correct inferred state from remaining models
   - This follows the existing model uninstall pattern of "clean everything and let the system rebuild"
