Create the `backend/app/views/` module with three files:

**`__init__.py`**: Empty init file.

**`registry.py`**: Extensible renderer registry.
- Create a module-level dict `RENDERER_REGISTRY` pre-populated with three built-in renderer types: `{"table": {"type": "table", "template": "browser/table_view.html"}, "card": {"type": "card", "template": "browser/card_view.html"}, "graph": {"type": "graph", "template": "browser/graph_view.html"}}`. Each entry has a `type` key and `template` key pointing to the Jinja2 template path.
- Provide a `register_renderer(type_str: str, handler: dict)` function that adds an entry to `RENDERER_REGISTRY`. The `handler` dict must contain at minimum `type` (str) and `template` (str, the Jinja2 template path). This allows Mental Models to register custom renderer types (e.g., timeline, kanban, calendar) at model installation time.
- Provide a `get_registered_renderers() -> dict` function returning the current registry contents (built-in + model-contributed).
- The ModelService calls `register_renderer()` during model installation for any custom renderers the model declares. Models declare custom renderers in their views JSONLD using `sempkm:customRenderer` entries with `sempkm:rendererType` and `sempkm:templatePath`. The ViewSpecService invokes `get_registered_renderers()` when resolving which renderer to use for a view spec. The `view_menu.html` template and view spec execution pipeline dynamically support whatever types are in the registry.

**`service.py`**: ViewSpecService class following the same pattern as ShapesService (see `app/services/shapes.py`).
- Define a `ViewSpec` dataclass with fields: `spec_iri: str`, `label: str`, `target_class: str`, `renderer_type: str`, `sparql_query: str`, `columns: list[str]` (default empty), `sort_default: str` (default ""), `card_title: str` (default ""), `card_subtitle: str` (default ""), `source_model: str` (default "").
- Constructor takes `TriplestoreClient` and `LabelService`.
- `get_all_view_specs() -> list[ViewSpec]`: Query the model registry for installed model IDs (same pattern as `ShapesService._fetch_shapes_graph`), then for each model build `FROM <urn:sempkm:model:{id}:views>` clauses. Execute a SPARQL SELECT querying `?spec a <urn:sempkm:vocab:ViewSpec>` with all view spec properties (targetClass, rendererType, sparqlQuery, columns, sortDefault, cardTitle, cardSubtitle). Return parsed ViewSpec dataclasses. Use rdfs:label for the spec label.
- `get_view_specs_for_type(type_iri: str) -> list[ViewSpec]`: Filter `get_all_view_specs()` by target_class matching type_iri.
- `execute_table_query(spec: ViewSpec, page: int, page_size: int, sort_col: str, sort_dir: str, filter_text: str) -> dict`: Execute the view spec's SPARQL query with pagination and sorting. CRITICAL: Use `scope_to_current_graph()` from `app/sparql/client` to inject `FROM <urn:sempkm:current>` into the spec's SPARQL query before execution. Use a two-phase approach per Research Pitfall 5: (1) wrap the original WHERE clause in a subquery with `SELECT DISTINCT ?s ... LIMIT/OFFSET` for pagination, (2) retrieve full properties for the paged subjects. Return dict with keys: `rows` (list of dicts), `total` (int), `page` (int), `page_size` (int), `total_pages` (int), `columns` (list of column names from spec.columns parsed by splitting on comma). If `filter_text` is provided, inject a `FILTER(REGEX(STR(?title), "{filter_text}", "i"))` into the WHERE clause (match against the first column variable). If `sort_col` and `sort_dir` are provided, add `ORDER BY ASC/DESC(?sort_col)` to the query.
- For the count query: wrap the spec's WHERE clause in `SELECT (COUNT(DISTINCT ?s) AS ?total) WHERE { ... }` with the same FROM scoping.

**Wire into app:**
- Add `get_view_spec_service` dependency to `dependencies.py` following the existing pattern (reads from `request.app.state.view_spec_service`).
- In `main.py` lifespan, after ShapesService creation, create `ViewSpecService(client, label_service)` and store as `app.state.view_spec_service`.

**`backend/app/views/router.py`**: Create view router with `prefix="/browser/views"`.
- `GET /browser/views/list/{type_iri:path}` — returns available view specs for a type (table, card, graph) as an HTML partial. Lists all view specs from ViewSpecService grouped by renderer type. Each entry is a link that opens the view via htmx. This is the "view menu" referenced in CONTEXT.md decisions.
- `GET /browser/views/table/{spec_iri:path}` — renders the table view for a given view spec IRI. Query params: `page` (default 1), `page_size` (default 25), `sort` (default from spec.sort_default), `dir` (default "asc"), `filter` (default ""). Calls `ViewSpecService.execute_table_query()` and renders `table_view.html`. The response is an htmx partial (renders into `#editor-area`). Resolve column header labels via LabelService -- column variables from the spec map to property IRIs that need human-readable names. Each row's `?s` value (the object IRI) should link to open the object in a tab via `onclick="openTab('iri', 'label')"` per existing pattern.
- Import the router in `main.py` as `views_router` and include it BEFORE browser_router (since browser_router has `/browser` prefix, views_router with `/browser/views` prefix needs to match first -- OR simply use the same browser prefix with sub-paths. Actually, since the views router has prefix `/browser/views` which is more specific than `/browser`, include it before browser_router).

**`backend/app/templates/browser/view_toolbar.html`**: Shared toolbar partial.
- Contains: view type switcher (icons for table/card/graph that link to the same type's other view specs), a text filter input with `hx-get` triggering on keyup with 300ms debounce (`hx-trigger="keyup changed delay:300ms"`), and current view label.
- Filter input sends its value as `filter` query param to the current view endpoint.

**`backend/app/templates/browser/table_view.html`**: Server-rendered table.
- Extends or includes `view_toolbar.html` at the top.
- Renders a `<table class="view-table">` with `<thead>` containing sortable column headers. Each header is an `<a>` with `hx-get` pointing to the same view endpoint with `sort={col}&dir={toggled_dir}&page=1&filter={current_filter}`, targeting `#editor-area`. Show sort indicator arrows (up/down unicode arrows) on the active sort column.
- Table `<tbody>` renders rows from the `rows` data. First column (object title/label) is clickable via `onclick="openTab('iri', 'label')"` to open the object in a tab (per existing pattern in workspace.js).
- Each `<th>` and corresponding `<td>` should have a `data-col="{{ col }}"` attribute so column-prefs.js can toggle visibility via CSS `display: none`.
- Includes `pagination.html` at the bottom.

**Column visibility customization (`frontend/static/js/column-prefs.js`)**:
- Provides a `ColumnPrefs` object with methods:
  - `getVisibleColumns(typeIri)`: Returns the saved column visibility/order array from `localStorage.getItem('col-prefs:' + typeIri)`. If none saved, returns null (meaning "show all defaults").
  - `saveColumnPrefs(typeIri, columns)`: Saves the array of `{col, visible, order}` objects to localStorage.
  - `applyColumnPrefs(typeIri)`: Reads prefs, hides `data-col` cells for hidden columns by toggling a `.col-hidden` CSS class, reorders visible columns via CSS `order` property.
  - `openColumnSettings(typeIri, allColumns)`: Opens a dropdown/popover UI anchored to a "Columns" gear icon button in the view toolbar. The dropdown shows a checklist of all available columns with checkboxes for visibility and drag handles for reorder (or simple up/down arrow buttons for ordering). On change, calls `saveColumnPrefs` and `applyColumnPrefs` to update the table immediately.
- The "Columns" gear icon button is added to `view_toolbar.html` (next to the filter input), visible only for table views.
- On table view load, call `ColumnPrefs.applyColumnPrefs(typeIri)` to restore saved preferences.
- Add `<script src="/static/js/column-prefs.js"></script>` to `base.html` after workspace.js.
- Add CSS in `views.css`: `.col-hidden { display: none; }`, `.column-settings-dropdown` styled as an absolute-positioned panel with checklist items, z-index above table.

**`backend/app/templates/browser/pagination.html`**: Reusable pagination partial.
- Shows: "Page X of Y (Z total)" with Previous/Next links using `hx-get` with updated page param. Also shows numbered page links for up to 7 pages (with ellipsis for large page counts). Preserves current sort and filter params in all pagination links.
- **Jump-to-page input**: Include a small number input (`<input type="number" min="1" max="{{ total_pages }}" class="jump-to-page">`) with a "Go" button. On submit (or Enter keypress), it triggers an `hx-get` to the current view endpoint with `page={entered_value}` and preserves current sort/filter params. Style the input as a compact inline element next to the page numbers. Add `.jump-to-page` CSS in `views.css`: width 50px, text-align center, same height as pagination buttons.

**`frontend/static/css/views.css`**: Styles for the view system.
