**Create `frontend/static/js/theme.js`**:
- IIFE structure, localStorage key `sempkm_theme`
- `getStoredPreference()` returns stored value or 'system'
- `resolveTheme(preference)` returns 'light' or 'dark' (system reads matchMedia once at load)
- `applyTheme(resolved)` sets `data-theme` attribute, updates `.theme-btn` active states, removes `.no-transition` class via `requestAnimationFrame`, dispatches `sempkm:theme-changed` CustomEvent with `{theme, preference}` detail
- `window.setTheme(preference)` stores to localStorage and calls applyTheme
- On load: call `applyTheme(resolveTheme(getStoredPreference()))` to wire up active states (the anti-FOUC script already set data-theme visually)

  <action>
Systematically replace every hardcoded hex color (`#xxx`, `#xxxxxx`), `rgb()`, and `rgba()` value across all four CSS files with the appropriate `var(--color-*)` token reference from theme.css. This is the bulk migration that makes dark mode actually work.

**Rules:**
1. Remove the duplicate `:root` color token declarations from `style.css` (they now live in theme.css). Keep non-color tokens (`--font-sans`, `--font-mono`, `--radius`, `--max-width`) in style.css `:root`.
2. Every `background`, `background-color`, `color`, `border-color`, `border`, `box-shadow`, `outline`, `fill`, `stroke` value that contains a hex/rgb/rgba must become a token reference.
3. Map hardcoded values to semantic tokens:
   - `#ffffff` / `#fff` → `var(--color-surface)` (or `--color-bg` for page bg)
   - `#f4f5f7` / `#f8f9fb` / `#eef0f4` → `var(--color-surface-raised)` or `--color-surface-recessed`
   - `#1a1a2e` / `#333` → `var(--color-text)`
   - `#666` / `#777` → `var(--color-text-muted)`
   - `#999` / `#aaa` → `var(--color-text-faint)`
   - `#e0e0e0` / `#ddd` / `#eee` → `var(--color-border)` or `--color-border-subtle`
   - `#2d5a9e` → `var(--color-primary)`
   - `#1e3f7a` → `var(--color-primary-hover)`
   - `#2a8a4a` / green variants → `var(--color-success)` or `--color-success-bg`
   - `#c0392b` / red variants → `var(--color-error)` or `--color-error-bg`
   - `#d4a017` / yellow variants → `var(--color-warning)` or `--color-warning-bg`
   - `#f5f5f5` → `var(--color-code-bg)`
   - Any `rgba(0,0,0,...)` light shadows → `var(--shadow)` or keep as `rgba()` with token-derived approach
4. For `rgba()` values used as hover overlays (e.g., `rgba(0,0,0,0.04)`), use `var(--color-surface-hover)`.
5. For color values used in `box-shadow`, replace with `var(--shadow)` or `var(--shadow-elevated)` tokens.
6. If a color value does not map cleanly to an existing token, add a new token to theme.css (e.g., `--color-badge-bg`, `--color-input-bg`).
7. After migration, grep all four CSS files for any remaining `#` hex patterns — the ONLY hex values should be in comments or in theme.css token definitions.

**File-by-file approach:**
- `style.css` (~29 hex values, ~15 remaining after tokens removed): sidebar colors, nav styles, layout chrome
- `workspace.css` (~181 hex values, largest effort): tab bar, editor area, toolbar, Split.js gutters, command palette overrides
- `forms.css` (~56 hex values): form fields, validation states, buttons, autocomplete
- `views.css` (~119 hex values): table, card, graph views, pagination, filter styles
