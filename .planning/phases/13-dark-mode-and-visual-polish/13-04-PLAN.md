  <name>Task 1: Fix Ctrl+K command palette in Firefox and pin ninja-keys version</name>
  <files>frontend/static/js/workspace.js, backend/app/templates/base.html</files>
  <action>
In `frontend/static/js/workspace.js`, inside the `initKeyboardShortcuts()` function (around line 482), add a Ctrl+K / Cmd+K case BEFORE the closing `});` of the keydown handler (after the Ctrl+E block, around line 530):

```javascript
// Ctrl+K / Cmd+K: Open command palette
if (mod && e.key === 'k') {
  e.preventDefault();
  var ninja = document.querySelector('ninja-keys');
  if (ninja) {
    ninja.open();
  }
}
```

This is needed because hotkeys-js (used internally by ninja-keys) silently ignores keydown events when focus is on INPUT/TEXTAREA/SELECT elements. Without our own handler calling preventDefault(), Firefox intercepts the unhandled Ctrl+K to focus its URL/search bar. By catching it at the document level first with preventDefault(), we guarantee the palette opens regardless of focus state or browser.

In `backend/app/templates/base.html`, pin the ninja-keys CDN import to a specific version. Change:
```html
<script type="module" src="https://unpkg.com/ninja-keys?module"></script>
```
to:
```html
<script type="module" src="https://unpkg.com/ninja-keys@1.2.2?module"></script>
```
(Use version 1.2.2 which is the current latest stable. Pinning prevents future breaking changes from unpinned CDN resolution.)

  <files>frontend/static/css/views.css, frontend/static/css/workspace.css</files>
  <action>
**Fix 1 — Remove view-tab border-left bleed** in `frontend/static/css/views.css`:

Find the `.workspace-tab.view-tab` rule (around line 651) and remove the entire `border-left: 2px solid var(--color-primary)` declaration. Also find `.workspace-tab.view-tab.active` (around line 655) and remove the `border-left-color: var(--color-primary)` declaration. These rules cause a permanent teal/blue left border on all view tabs. The active tab accent should come ONLY from the bottom border defined in workspace.css (`border-bottom: 2px solid var(--color-accent)`).

After removal, the selectors should either be empty (and can be removed entirely) or contain only non-border-left properties. If both selectors become empty, delete them.

**Fix 2 — Add padding-top to tab bar for border-radius headroom** in `frontend/static/css/workspace.css`:

Find `.tab-bar-workspace` (around line 274). Add `padding-top: 4px;` to the rule. Currently `overflow-y: hidden` combined with `align-items: flex-end` clips the top 4px border-radius of tabs because there is zero headroom above the tab tops and the clip boundary. Adding 4px padding-top gives the rounded corners room to render above the overflow clip edge.

**Fix 3 — Add distinctive borders to card faces** in `frontend/static/css/views.css`:

Find the `.flip-card-front, .flip-card-back` rule (around line 715). Add `border: 1px solid var(--color-border);` to this rule. The cards currently rely only on `box-shadow: var(--shadow)` which is too faint (8% opacity in light mode, 30% in dark mode).

Find the `.card-focus-front, .card-focus-back` rule (around line 763). Add `border: 1px solid var(--color-border);` to this rule as well. This ensures both the inline flip cards AND the focus portal (enlarged) cards have visible borders.

The `--color-border` token is already defined in theme.css for both light (#e0e0e0) and dark (#3e4452) modes, so the borders will automatically adapt to the active theme.
