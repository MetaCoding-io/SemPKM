**SPARQL router -- promote/demote endpoints (router.py):**

1. `POST /sparql/saved/{query_id}/promote` -- Accepts JSON body: `{ "display_label": str, "renderer_type": str }`. Creates a `PromotedQueryView` row for the current user's saved query. Only the query owner can promote. If already promoted, return 409 Conflict. Returns 201 with `{ "id": uuid, "spec_iri": "urn:sempkm:user-view:{id}" }`. Validates `renderer_type` is one of "table", "card", "graph".

2. `DELETE /sparql/saved/{query_id}/promote` -- Demotes a promoted view. Deletes the `PromotedQueryView` row. Only the query owner can demote. Returns 204. The underlying saved query remains intact.

3. `GET /sparql/saved/{query_id}/promotion` -- Check if a saved query is promoted. Returns `{ "promoted": bool, "view_id": uuid|null, "spec_iri": str|null, "display_label": str|null, "renderer_type": str|null }`. Owner-only.

**ViewSpecService integration (views/service.py):**

1. Add `_extract_select_var_names(sparql_query: str) -> list[str]` as a module-level function:
   - Use regex to match `SELECT (DISTINCT)? ... WHERE` (case-insensitive, DOTALL)
   - Handle `SELECT *` by returning empty list
   - Extract aliases: `AS ?alias` -> use alias name
   - Extract direct variables: `?var` (outside parentheses)
   - Return variable names without the `?` prefix

2. Add `async def get_user_promoted_view_specs(self, user_id: uuid.UUID, db) -> list[ViewSpec]` method to `ViewSpecService`:
   - Query `PromotedQueryView` joined with `SavedSparqlQuery` where `PromotedQueryView.user_id == user_id`
   - For each result, build a `ViewSpec` with:
     - `spec_iri = f"urn:sempkm:user-view:{pv.id}"`
     - `label = pv.display_label`
     - `target_class = ""` (user views don't target a specific class)
     - `renderer_type = pv.renderer_type`
     - `sparql_query = sq.query_text`
     - `columns = _extract_select_var_names(sq.query_text)`
     - `source_model = "user"`
   - Do NOT cache these results (per Research pitfall 1 -- keep user specs out of TTL cache)
   - Import needed: `from sqlalchemy.ext.asyncio import AsyncSession`, `from sqlalchemy import select`, `from app.sparql.models import PromotedQueryView, SavedSparqlQuery`

3. Extend `get_view_spec_by_iri()` to accept an optional `user_id` and `db` parameter. If spec_iri starts with `"urn:sempkm:user-view:"`, query `PromotedQueryView` from SQLite instead of searching cached model specs. Return the constructed `ViewSpec` or None.

4. **CRITICAL -- Promoted view table rendering (source_model == "user"):** Modify `execute_table_query()` to handle promoted views differently from model views. The existing code (lines 316-328) deduplicates rows by the `?s` binding and builds each row starting with `row = {"s": subject}`. Promoted user views will NOT have a `?s` variable -- their queries use arbitrary SELECT variables. When `spec.source_model == "user"`:
   - **Skip the `?s`-based deduplication entirely.** Do not filter by `seen_subjects`. Render ALL result rows as-is.
   - Build each row from the spec's columns directly: `for col in spec.columns: row[col] = b.get(col, {}).get("value", "")`. Do NOT prepend an `"s"` key.
   - For URI values in the results, render them as plain `<a href>` links (not IRI pills). This matches the user decision "plain links for IRIs" from CONTEXT.md. The template rendering side handles this -- ensure the `table_view.html` template checks `source_model` and renders URI cells as `<a href="{value}">{label_or_value}</a>` instead of the IRI pill component used for model views.
   - See RESEARCH.md Pitfall 3 for full rationale.
