**Ch 25: WebID Profiles (~150 lines):**
Follow the new chapter template from RESEARCH.md.

1. **Introduction**: What WebID is — a URI that identifies you on the web and resolves to a machine-readable profile. Why it matters for decentralized identity and fediverse verification.
2. **Your WebID URI**: Each user gets a WebID at `https://your-instance/users/username#me`. The `#me` fragment identifies the person (vs. the profile document).
3. **Viewing Your Profile**: Navigate to Settings > Identity (or the profile URL). Describe the HTML profile page showing name, links, public key.
4. **Content Negotiation**: The profile URL serves different formats based on the request:
   - Browser (Accept: text/html) -> HTML page
   - RDF client (Accept: text/turtle) -> Turtle document
   - JSON client (Accept: application/ld+json) -> JSON-LD document
   - `?format=turtle` or `?format=jsonld` query parameter fallback
5. **Fediverse Verification (rel="me")**: Add your Mastodon/fediverse profile URL in Settings > Identity. Your WebID page includes `rel="me"` links. Add your WebID URL to your Mastodon profile to complete bidirectional verification.
6. **Cryptographic Keys**: Each user has an Ed25519 key pair. The public key is published in the profile document. Used for future signing and federation features.
7. **For Advanced Users**: Explain the RDF structure (FOAF, schema.org), how the key is serialized (multibase), and how to verify the profile with curl.

Navigation footer: Previous -> Ch 24, Next -> Ch 26.

**Ch 26: IndieAuth (~150 lines):**
Follow the new chapter template.

1. **Introduction**: What IndieAuth is — an OAuth2-based authentication protocol using your URL as your identity. Lets you sign into IndieWeb-compatible services with your SemPKM URL.
2. **How It Works**: Brief description of the flow — client discovers your endpoints, redirects to SemPKM for authorization, you see a consent screen, approve, client gets a token.
3. **Discovery**: Your WebID profile page includes `rel="indieauth-metadata"` link. Clients find the authorization and token endpoints from the metadata document.
4. **The Consent Screen**: When an app requests access, SemPKM shows a consent screen displaying the app name, requested scopes, and your identity. You can approve or deny.
5. **Managing Authorized Apps**: Settings > Identity shows authorized applications. You can revoke access.
6. **Supported Scopes**: List the scopes the IndieAuth provider supports (profile, create, update, etc.).
7. **For Advanced Users**: Authorization endpoint URL, token endpoint URL, PKCE requirement (S256), how to test with IndieAuth tools. Explain this is for app developers integrating with SemPKM.
8. **Troubleshooting**: Common issues (APP_BASE_URL misconfigured, CORS, cookie issues).

Navigation footer: Previous -> Ch 25, Next -> Appendix A.

Use document conventions throughout. Basic PKM examples where applicable.
