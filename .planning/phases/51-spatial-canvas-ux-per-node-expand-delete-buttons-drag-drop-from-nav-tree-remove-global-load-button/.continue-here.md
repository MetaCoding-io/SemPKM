<decisions_made>

- **Side-channel pattern for dockview**: Instead of fighting HTML5 DnD inside dockview, store drag payload in `window.__canvasDragPayload` and use `dragend` as fallback when `drop` never fires
- **Capture-phase listeners on document**: Canvas drag handlers registered on `document` with `capture: true` and use `isOverCanvas()` bounding-rect check instead of relying on event.target (which dockview overlays can steal)
- **Model-agnostic body fetch**: SPARQL `FILTER(STRENDS(STR(?p), ":body"))` matches any model's body predicate, not just `urn:sempkm:body`
- **Async body fetch**: Body is fetched after node creation (non-blocking), node appears immediately with empty markdown, re-renders when body arrives
</decisions_made>

<context>
The root cause is clear: dockview intercepts HTML5 drag-drop events inside its panels. The `dragend` fallback works because `dragend` always fires on the source element regardless of what happened at the target. The Chrome issue is likely that `event.dataTransfer.types` doesn't include custom MIME types during `dragover` in Chrome's protected mode — this means our guard `types.indexOf('text/iri') === -1` returns early without setting `lastDragOverCanvas`, so the `dragend` fallback has no position to use.

Quick fix to try: in `onDragOver`, instead of checking for `text/iri` in types, check `window.__canvasDragPayload` existence. This is reliable because the payload is set synchronously in `ondragstart` before any drag events fire.
</context>
