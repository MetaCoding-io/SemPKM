0. **frontend/nginx.conf** — Update nginx configuration for auth:
   - Ensure `/api/` proxy passes cookies correctly: add `proxy_pass_header Set-Cookie;` and `proxy_set_header Cookie $http_cookie;` to the `/api/` location block
   - The existing proxy_pass to `http://api:8000/api/` handles auth endpoints since they're under `/api/auth/`
   - No new location blocks needed -- static HTML files are already served by the default `location /` block

1. **frontend/static/js/auth.js** — Auth page JavaScript (vanilla JS, no frameworks, matching existing dev console pattern):
   - `checkAuthStatus()`: Fetch GET /api/auth/status. If setup_mode is true and not on setup page, redirect to /setup.html. If setup is complete and not authenticated, redirect to /login.html. If authenticated, allow navigation.
   - `handleSetupForm()`: On setup.html form submit, POST /api/auth/setup with token (and optional email). On success, show success message and redirect to main page after 2 seconds. On error, show error message.
   - `handleLoginForm()`: On login.html form submit, POST /api/auth/magic-link with email. Show "Check your email for a login link" message. Note: for local instances with no SMTP, the login page should explain that only the owner session (from setup) works until SMTP is configured.
   - `handleVerifyToken()`: On page load, check URL for `?token=...` parameter. If present, POST /api/auth/verify with token. On success, redirect to main page. On error, show error.
   - `handleLogout()`: POST /api/auth/logout, then redirect to /login.html.
   - `handleInviteAccept()`: On invite.html, check URL for `?token=...`. POST /api/auth/verify-invitation (or similar endpoint). On success, redirect to main page.

2. **frontend/static/setup.html** — Setup wizard page:
   - Clean, minimal design matching existing dev console CSS (same style.css)
   - SemPKM branding at top
   - Heading: "Welcome to SemPKM"
   - Subtext: "Enter the setup token shown in your terminal to claim this instance as owner."
   - Form with one input field: "Setup Token" (text, autofocus)
   - Optional email field: "Email (optional)" with placeholder "owner@localhost"
   - Submit button: "Claim Instance"
   - Status/error message area below the form
   - Include auth.js and call checkAuthStatus() + handleSetupForm() on load

3. **frontend/static/login.html** — Login page:
   - Same clean design as setup page
   - Heading: "Sign in to SemPKM"
   - Form with email input and "Send Magic Link" button
   - Info note below: "A login link will be sent to your email. If this is a local instance without email configured, use the terminal setup flow."
   - If URL has `?token=...` query param (magic link callback), auto-verify the token
   - Status/error message area
   - Include auth.js and call appropriate handlers on load

4. **frontend/static/invite.html** — Invitation acceptance page:
   - Same design as login page
   - Heading: "You've been invited to SemPKM"
   - Auto-reads `?token=...` from URL
   - Shows "Verifying your invitation..." while processing
   - On success, shows "Welcome! Redirecting..." and redirects to main page
   - On error, shows error message with option to request a new invitation
   - Include auth.js

5. **frontend/static/css/style.css** — Add auth page styles (append to existing):
   - `.auth-container`: centered card layout (max-width 420px, margin auto, padding)
   - `.auth-form`: form styling (input fields, submit button matching existing form styles)
   - `.auth-message`: status message styling (success green, error red)
   - `.auth-info`: info text styling (muted, smaller font)
   - Keep consistent with existing dev console styling (same font, colors, spacing)
