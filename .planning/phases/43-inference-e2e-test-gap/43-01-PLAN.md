  <name>Task 1: Fix Literal filter in inference service</name>
  <files>backend/app/inference/service.py</files>
  <action>
In `service.py`, modify the step 6 BNode filter (lines 174-179) to also filter out triples with `rdflib.Literal` subjects or objects. Add `Literal` to the existing import from rdflib (check existing imports first -- `Literal` may already be imported).

Change the filter from:
```python
new_triples = {
    (s, p, o)
    for s, p, o in new_triples
    if not isinstance(s, BNode) and not isinstance(o, BNode)
}
```

To:
```python
new_triples = {
    (s, p, o)
    for s, p, o in new_triples
    if not isinstance(s, BNode) and not isinstance(o, BNode)
    and not isinstance(s, Literal) and not isinstance(o, Literal)
}
```

Update the log message on line 180 to say "after BNode/Literal filter" for clarity.

Why: owlrl's rdfs:domain/rdfs:range entailment produces triples like `"active" rdf:type xsd:string` where the subject is a Literal. The `_store_inferred_triples` method wraps all components as `<{value}>` (URI angle-bracket syntax), which is invalid for Literals and causes SPARQL 400 errors. Filtering Literals is correct because these are schema-level type assertions, not meaningful user-facing inference results.

Do NOT modify `_store_inferred_triples` itself -- the filter belongs at step 6 where BNodes are already filtered, before triples reach storage.
